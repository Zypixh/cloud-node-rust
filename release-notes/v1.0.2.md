# cloud-node-rust 1.0.2

本版本重点围绕 `TOA`、`SNI passthrough`、`HTTP/3` 稳定性和热路径性能做了收口。

## 重点更新

- 增加发送侧 TOA 方案骨架
  - 新增 `toa-sender/` Linux 内核模块目录，用于在边缘节点出站 SYN 上注入 `TCP option 254`
  - 主程序内置 sender 控制逻辑，不再依赖独立 daemon
  - `TOA` 只在配置 `toa.isOn=true` 时启用，关闭时完全走普通回源路径
  - 控制面下发 `toa.isOn=true` 后，主程序会在 Linux 上自动检查并尝试编译/加载 `cloud_toa_sender.ko`
  - sender 映射已升级为：
    - `local_port`
    - `backend_ip`
    - `backend_port`
    - `family`
  - 首次进入 TOA 链路时会先执行一次 `FLUSH`，降低脏映射残留风险

- 修正并增强 `@sni_passthrough`
  - 继续复用共享 HTTPS 端口
  - 回源严格尊重源站配置的 `host:port`
  - 增加 LB 缺失时的临时重建回退
  - 保留基础流量统计、回源流量统计和计费维度上报

- 优化 HTTP/3 bridge 稳定性
  - 请求体和响应体增加大小护栏
  - 响应侧改为流式回传，降低大响应内存峰值
  - 请求侧增加声明长度预拒绝和预分配，减少无意义缓冲

- 优化主请求热路径
  - 合并热路径配置读取，减少 `ConfigStore` 同步锁往返
  - 合并 `server + upstream` 查询
  - HTTPS ClientHello 预读改为固定缓冲，减少堆分配

- 为重型响应处理增加内存保护
  - `optimization`
  - `WebP`
  - `HLS playlist`
  - `HLS segment`
  - 超出阈值时会跳过处理或快速失败，避免大对象放大内存占用

## 兼容性说明

- `TOA` 当前属于 Linux 定向能力
  - 边缘节点需要加载 `cloud_toa_sender.ko`
  - 后端需要加载 `toa-main/toa.ko`
  - 非 Linux 环境下，如果开启 `toa.isOn=true`，主程序会明确报错

- `TOA` 发送侧代码已经接入主程序，但本次发布仍保留一个明确边界：
  - 当前仓库内已完成代码接入和主程序编译验证
  - 还未包含 Linux 端到端联调结果
  - GitHub Release 会附带 `toa-main` 和 `toa-sender` 的源码包，不会直接附带 `.ko` 二进制
