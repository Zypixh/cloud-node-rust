# cloud-node-rust 1.0.4

本文件记录 `1.0.4` 的发布说明。

## 修复与优化

- **重构 gRPC 与 WebSocket 的兼容性逻辑**
  - **支持联动**：恢复了“开启 WebSocket 自动激活 gRPC 支持”的业务逻辑。现在当站点开启 WebSocket 或 gRPC 任一开关时，系统都会进入全协议支持模式（`ctx.is_grpc = true`）。
  - **智能协议识别**：在回源阶段引入了精准识别机制。即使系统处于 gRPC 支持模式，也**仅在请求内容确认为 `application/grpc` 时**才会强制使用 HTTP/2 (H2) 回源。
  - **结果**：完美解决了 1.0.3 中由于 WebSocket 与 gRPC 联动导致普通 HTTP/1.1 WebSocket 请求被强制 H2 引起 `502` 的问题，实现了功能增强与向后兼容的统一。

- **优化上游错误处理逻辑**
  - **状态保留**：在 `fail_to_proxy` 阶段，对“真实上游 HTTPStatus”（如源站返回的 5xx）不再一律套用本地自定义错误页。
  - **区分故障**：此举旨在区分“源站真实的业务错误”与“代理层自身的连接故障（如连接拒绝、超时）”。
  - **行为一致性**：源站返回的 5xx 状态码现在将直接透传给客户端，仅在代理层无法连接源站或握手失败时，才会触发本地配置的自定义 502/504 错误页。

- **代码质量与逻辑简化**
  - 修正了 `fail_to_proxy` 中的日志打印描述，从 "failed to send custom error response" 调整为更通用的 "failed to send error response"。

## 自动构建说明

- **关于构建耗时（约 15 分钟）的说明**
  - **高性能优化开关**：为了追求极致的运行时性能，我们的自动构建开启了 `LTO (Link Time Optimization) = fat` 以及 `codegen-units = 1`。这些设置会极大增加链接阶段的计算量，但能显著提升二进制文件的执行效率。
  - **多架构并发**：CI 同时为 x86-64 (v2/v3/v4) 和 ARM64 (Generic/Neoverse-N1) 多个目标架构生成优化包。
  - **缓存状态**：经检查，GitHub Actions 的 `rust-cache` 已正确启用。但在处理 Docker 容器内的 Legacy (glibc 2.17) 构建时，由于需要重新安装工具链，耗时相对较长。
