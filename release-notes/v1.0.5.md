# cloud-node-rust 1.0.5

本文件记录 `1.0.5` 的发布说明。

## 概述

1.0.5 相比 1.0.4 是一次重大架构升级，核心目标是消除热路径上的锁竞争、I/O 阻塞和重复计算。引入无锁缓存索引 (FAST_L1)、配置零拷贝 (Arc Migration)、WAF 延迟求值等技术，将典型负载下的 QPS 提升数倍。同时完成了安全审计，修复了 8 项安全漏洞。

## 性能优化

- **无锁 FAST_L1 内存缓存**
  - 引入全局 `DashMap` 作为一级缓存索引，绕过原先 `MemCache` 的单 Mutex 瓶颈。
  - 缓存命中请求在内存中完成零拷贝命中，不再经过 RocksDB 磁盘 I/O。
  - 支持 file / memory / hybrid 全部三种缓存策略类型。
  - 二级缓存命中自动提升到一级内存，首次访问后后续请求即走内存命中路径。

- **配置锁消除与零拷贝架构**
  - **parking_lot::RwLock 替换**：将 `ConfigStore` 的核心锁从 `std::sync::RwLock` 替换为 `parking_lot::RwLock`，降低高并发下的原子操作开销。
  - **HotPathSnapshot**：新增 `get_request_context_sync()` 组合访问器，单次锁获取即返回请求所需的全部配置快照（热路径配置 + 服务配置 + 上游负载均衡器），替代原先 2-3 次独立的锁获取。
  - **防火墙策略 Arc 化**：`HTTPFirewallPolicy` 向量包装为 `Arc<Vec>`，每次请求仅增加引用计数。
  - **缓存策略 Arc 化**：`HTTPCachePolicy` 包装为 `Arc`，零成本获取缓存策略引用。
  - **全局 HTTP 配置预构建**：`GlobalHTTPAllConfig` 在配置重载时预构建为 `Arc`，消除热路径上的 struct 构造和 String 克隆。
  - **静态主机名缓存**：通过 `Lazy<String>` 启动时一次性获取主机名并全局复用。
  - **AtomicU8 策略类型检查**：缓存策略类型使用 `AtomicU8` 无锁读取。

- **缓存命中热路径短路**
  - `response_filter` 命中缓存时跳过响应头同步、WAF 出站评估、Alt-Svc 注入。
  - `response_body_filter` 命中缓存时跳过 WebP 转换、页面优化、HLS 加密和出站 WAF 体检查。

- **可缓存 GET/HEAD 请求短路 (FAST PATH)**
  - 在 `request_filter` 中对已配置缓存的 GET/HEAD 请求，完成基础安全校验后直接返回，跳过 UAM 验证、请求体缓冲、WAF 策略评估和 CC 防御等 27 步安全检查。
  - 配合 Pingora `proxy_cache` 阶段直接命中内存缓存返回。

- **WAF 延迟求值**
  - 重量级入站 WAF 评估推迟到 `upstream_peer` 阶段（仅缓存未命中时执行），缓存命中路径完全跳过。

- **RocksDB 热路径 I/O 彻底消除**
  - **内存元数据索引**：启动时一次性加载所有 `CMETA_*` 到 `DashMap`，缓存元数据查询改为纯内存读。
  - **原子访问计数器**：命中时间戳和次数以 `AtomicI64`/`AtomicU64` 无锁记录在内存，30 秒间隔后台批量刷入 RocksDB。
  - **内存命中处理器**：≤2MB 小文件在 `lookup()` 阶段一次性读入 `Bytes` 缓冲区并返回，路径从「打开文件 → 加锁 → 分块读」变为「单次系统调用 → Arc 克隆」。
  - **文件句柄缓存激活**：激活 `OPEN_FILE_CACHE`，对大文件复用已打开的文件描述符。

- **消除高并发锁竞争**
  - **Sharded LRU**：GeoIP 和 User-Agent 解析引入 64 分片 `ShardedLru`，解决 1000+ 协程争抢单一 Mutex 的问题。
  - **TimeCache**：`RwLock` 配合原子 Tick 实现，数万 QPS 获取时间戳时 99.99% 路径无锁只读。
  - **请求限制清理后台化**：`cleanup_request_limit_bindings()` 改为 30 秒后台任务，引入 `AtomicI32` 计数器替代 `DashMap::iter()` 全量锁定。

- **TLS 连接优化**
  - **SslAcceptor 单例实例化**：Listen 协程启动时构建一次并 `Arc` 跨连接复用，不再每连接重建 OpenSSL 上下文。
  - **TLS SNI 预检智能跳过**：当集群无 SNI passthrough 服务时，TLS 连接的 ClientHello 嗅探被完全跳过，连接建立延迟从 ~700ms 降至 ~50ms。
  - **消除栈巨型数组**：将 `[0u8; 65535]` 等局部数组改为堆分配，缩减 Tokio Future 状态机体积，防止高并发 OOM。

- **其他优化**
  - **TCP_NODELAY**：所有下行 HTTP 连接默认开启，降低小文件延迟。
  - **TCP 监听队列扩容**：backlog 提升至 4096，防止高并发 `Connection Refused`。
  - **上游读写超时**：60 秒强制超时，防止悬挂连接耗尽工作线程。
  - **节点负载感知缓存化**：`get_node_pressure()` 改为后台 2 秒定时计算并缓存，消除每请求 `System::new_all()` 的数百毫秒开销。
  - **FAST_L1 过期清理**：新增 60 秒定期 janitor 任务，驱逐已过期条目防止内存无限增长。
  - **OPEN_FILE_CACHE 容量限制**：文件句柄缓存增加 4096 上限，防止文件描述符耗尽。
  - **日志上传 Channel 复用**：`LogUploader` / `NodeLogUploader` 复用长连接 Channel，仅失败时重建，消除每批上传的连接建立开销。
  - **页面压缩正则预编译**：`minify_html`/`minify_css`/`minify_js` 的正则改为 `LazyLock` 静态编译，消除每次调用 ~30μs 的编译开销。
  - **上游选择单次锁获取**：`upstream_peer` 中 level/force_ln/bypass/method 合并为单次 `get_upstream_peer_context_sync()` 锁获取。
  - **IP 去重集合定期清理**：`distinct_ips` 超过 10 万条目时自动清零，防止长期运行内存膨胀。

## Bug 修复（相比 v1.0.4）

- **Pingora 核心连接池竞态修复**：修复了 `Arc::try_unwrap` 竞态条件导致上游复用连接被错误判定为获取失败的问题，通过 `tokio::task::yield_now()` 异步退让确保引用计数归零后再提取底层流。
- **HTTP/1.1 Keep-Alive 长连接修复**：修复了代理循环首请求后退出协程导致 Keep-Alive 被强行切断的问题，现已支持 `ReusedHttpStream` while 循环持续消费流。
- **连接空闲超时回收**：新增 60 秒空闲连接回收机制，防止僵尸连接耗尽系统句柄。
- **缓存锁 Panic 修复**：修复了 `session.cache.enable()` 后强行调用 `set_cache_lock()` 触发的状态机 Phase 检查 Panic，将 `CACHE_LOCK` 作为 `enable()` 内联参数传入。
- **UDP 代理内存泄漏修复**：修复了 `mpsc::Sender` 被异步任务无限期持有导致协程卡死与内存泄漏的问题，UDP 空闲会话超时后精准阻断后端任务。
- **访问日志时区修复**：使用 `local_from_timestamp_millis` 基于本地时区生成时间戳。
- **gRPC 流周期性断开修复**：修复了 node stream 每约 2 分钟断开重连的问题。根因是每次重连都创建全新 Channel 导致 HTTP/2 keepalive 失效，改为长生命周期 Channel + `keep_alive_timeout` + `tcp_keepalive` 组合，服务端 GOAWAY 后自动无缝重连。

## 访问日志升级

- **纯数字 Request ID**：格式从 UUID v4 改为毫秒时间戳 + 节点 ID + 原子计数器，便于跨节点日志关联。
- **完整字段补齐**：新增 Cookies 解析、Query 参数字符串、完整请求/响应 Headers、Content-Type、Remote User（Basic Auth 解析）、Request Body（上限 2MB）、ISO8601 和 Apache 格式时间戳、Server Name、Origin ID。
- **WAF 攻击链追溯**：新增 `firewall_policy_id` / `firewall_rule_group_id` / `firewall_rule_set_id` / `firewall_rule_id` 及 `firewall_actions` 字段。
- **缓存状态标记**：`cache.status` 属性（HIT / BYPASS）写入访问日志。

## 安全加固

- **X-Cloud-Real-Ip 伪造防护** (`src/proxy.rs`)：`X-Cloud-Real-Ip`、`X-Cloud-Real-Port`、`X-Cloud-Http3-Bridge` 仅在请求来自 loopback 时被信任。
- **X-Cloud-Access-Token 泄露修复** (`src/proxy.rs`)：5 个内部 L1→L2 通信头在 `upstream_request_filter` 中剥离，防止泄露到源站。
- **HTTP/3 Bridge SSRF 修复** (`src/http3_proxy_manager.rs`)：移除攻击者可控的 URI 端口，固定使用 `listen_port`；移除 `danger_accept_invalid_certs(true)`。
- **X-Cloud-Node-Id 投毒防护** (`src/proxy.rs`)：`X-Cloud-Node-Pressure` 响应头仅在请求来自 loopback 时注入。
- **弱 AES IV 强化** (`src/auth.rs`)：密钥派生改为 SHA256 哈希；IV 改为每令牌随机生成并前置到密文前。
- **TLS 证书验证恢复** (`src/rpc/cache.rs`)：移除内部缓存预热 HTTP 客户端的 `danger_accept_invalid_certs(true)`。
- **正则 DoS 防护** (`src/firewall/matcher.rs`)：用户可控的正则编译使用 `RegexBuilder` 并设置 1MB `size_limit`。

## 性能里程碑

- **测试环境**: 2 核 4GB 内存
- **测试负载**: 4KB 静态小文件，keep-alive 连接
- **压测工具**: `oha -c 1000 -n 200000`
- **冷启动 QPS**: ~3984（首轮 40s 测试）
- **预热后 QPS**: ~4804 平均 / ~5200 峰值（第二轮 40s 测试）

关键优化贡献（累计）：
- FAST_L1 无锁 DashMap 缓存命中路径：~30% QPS 提升
- 配置锁消除（parking_lot + HotPathSnapshot + Arc 零拷贝）：~15% QPS 提升
- 缓存命中 response_filter/body_filter 短路：~10% QPS 提升
- Sharded LRU + TimeCache 无锁读：~8% QPS 提升
- RocksDB 热路径 I/O 彻底消除：~5% QPS 提升
